# 3.3 RAG Pipeline

## 1 技术简介

**RAG Pipeline**（检索增强生成流水线）是构建完整RAG系统的核心框架，它将文档加载、分块、向量化存储、检索和生成等环节有机整合，形成端到端的问答系统。RAG（Retrieval-Augmented Generation）通过结合外部知识库检索与大语言模型的生成能力，有效解决了纯LLM在知识时效性和幻觉问题上的局限。

RAG Pipeline的核心思想是：当用户提出问题时，系统首先从知识库中检索相关文档，然后将这些文档作为上下文提供给大语言模型，生成基于真实知识的回答。这种架构使得LLM能够：
- 访问最新、最专业的信息
- 生成可追溯、可验证的回答
- 减少幻觉和错误信息的产生

一个完整的RAG Pipeline通常包括以下关键环节：文档加载、文本分块、向量嵌入、向量存储、用户查询、向量检索、结果排序、上下文拼接、LLM生成。

---

## 2 技术背景

### 2.1 大语言模型的局限性

纯大语言模型虽然具备强大的语言理解和生成能力，但在实际应用中面临以下挑战：
- **知识截止**：模型训练数据有截止日期，无法获取最新信息
- **幻觉问题**：模型可能生成看似合理但实际错误的内容
- **专业领域知识不足**：通用模型在特定领域可能缺乏深度知识
- **长尾知识**：模型对低频、长尾信息记忆不完整

### 2.2 RAG的诞生与发展

RAG技术最初由Facebook AI Research在2020年提出，最初形态是将检索结果直接拼接在提示词中。随着向量数据库、embedding技术和大语言模型的发展，RAG逐渐演变成一个完整的系统工程，涌现出多种优化技术，如混合检索、查询重写、重排序、GraphRAG等。

### 2.3 RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| **知识更新** | 动态更新，无需重新训练 | 需要重新训练 |
| **可解释性** | 可追溯到源文档 | 知识编码在模型参数中 |
| **计算成本** | 检索+生成 | 主要是训练成本 |
| **适用场景** | 知识库频繁变化 | 任务模式固定 |

## 3 核心概念

### 3.1 文档加载（Document Loading）

从各种数据源（文本文件、PDF、网页、数据库等）加载文档。LangChain提供了多种文档加载器，如TextLoader、PyPDFLoader、WebBaseLoader等。

### 3.2 文本分块（Text Chunking）

将长文档切分成较小的文本块，是RAG系统中的关键预处理步骤。合理的分块策略能够提高检索精度和上下文完整性。

### 3.3 向量嵌入（Embedding）

将文本转换为向量表示的过程。Embedding模型将语义相近的文本映射到向量空间中相近的位置，使得可以通过向量相似度检索相关内容。

### 3.4 向量存储（Vector Store）

存储文档向量表示的数据库，支持高效的向量相似度检索。常见的向量数据库包括FAISS、Chroma、Milvus、Qdrant、Pinecone等。

### 3.5 检索（Retrieval）

根据用户查询从向量数据库中检索相关文档。检索质量直接影响最终生成效果，可以使用向量相似度检索、关键词检索或混合检索等方式。

### 3.6 生成（Generation）

将检索到的相关文档与用户问题一起拼接成提示词，调用大语言模型生成回答。

---

## 4 工作原理

### 4.1 离线流程（索引构建）

1. **文档加载**：从各类数据源加载原始文档
2. **文本预处理**：清洗文本，去除无关信息
3. **文本分块**：按照策略将文档切分成文本块
4. **向量化**：使用Embedding模型将文本块转换为向量
5. **向量存储**：将向量和原始文本存储到向量数据库
6. **构建索引**：为向量数据库建立索引以加速检索

### 4.2 在线流程（问答）

1. **接收查询**：用户输入问题
2. **查询向量化**：将用户问题转换为向量
3. **相似度检索**：在向量数据库中检索最相似的文档
4. **结果排序**：对检索结果进行排序和过滤
5. **上下文构建**：将相关文档拼接成上下文
6. **提示词构建**：将上下文和问题组合成完整的提示词
7. **LLM生成**：调用大语言模型生成回答
8. **结果返回**：将生成的回答返回给用户

### 4.3 两阶段检索

1. **粗排阶段**：使用高效的向量检索或BM25快速筛选出候选集（如Top-100）
2. **精排阶段**：使用更精确的模型（如Cross-Encoder）对候选集进行重排序，最终输出Top-K

## 5 技术实现

### 5.1 LangChain RAG组件

LangChain提供了完整的RAG组件库：
- **Document Loaders**：TextLoader、PyPDFLoader、DocxLoader等
- **Text Splitters**：RecursiveCharacterTextSplitter、CharacterTextSplitter等
- **Embeddings**：OpenAIEmbeddings、HuggingFaceEmbeddings等
- **Vector Stores**：FAISS、Chroma、Milvus等
- **Retrievers**：VectorStoreRetriever、EnsembleRetriever等
- **Chains**：RetrievalQA、ConversationalRetrievalChain等

### 5.2 Embedding模型选择

- **英文为主**：OpenAI text-embedding-ada-002、sentence-transformers/all-MiniLM-L6-v2
- **中文为主**：BAAI/bge-small-zh、BAAI/bge-base-zh
- **多语言**：BAAI/bge-m3、OpenAI text-embedding-3-large

### 5.3 向量数据库选择

- **轻量级**：FAISS（适合本地小规模数据）
- **开箱即用**：Chroma（Python原生，部署简单）
- **生产级**：Milvus、Qdrant（分布式架构，支持大规模数据）
- **云服务**：Pinecone、Azure AI Search（托管服务）

## 6 Python代码实现与逐行解析

```python
# 导入LangChain核心组件，构建完整的RAG Pipeline
# FAISS: Facebook AI Similarity Search，高效的向量相似度搜索库
from langchain.vectorstores import FAISS
# OpenAIEmbeddings: OpenAI的文本嵌入模型，将文本转为向量
from langchain.embeddings import OpenAIEmbeddings
# RecursiveCharacterTextSplitter: 递归文本分割器，保持语义完整性
from langchain.text_splitter import RecursiveCharacterTextSplitter
# TextLoader: 文本文件加载器
from langchain.document_loaders import TextLoader
# RetrievalQA: 检索问答链，整合检索和生成流程
from langchain.chains import RetrievalQA
# OpenAI: OpenAI的大语言模型接口
from langchain.llms import OpenAI

def rag_pipeline_example():
    """
    完整的RAG流程示例
    
    RAG = Retrieval + Generation
    完整的流程包含7个步骤：
    1. 加载文档 -> 2. 分块 -> 3. 生成Embedding -> 4. 存储到向量库
    5. 检索相关文档 -> 6. 构建Prompt -> 7. LLM生成回答
    """
    
    # 1. 加载文档
    # 使用TextLoader加载example.txt文件，指定UTF-8编码
    # loader.load()返回Document对象列表，每个Document包含page_content和metadata
    loader = TextLoader("example.txt", encoding="utf-8")
    documents = loader.load()
    
    # 2. 文档分块
    # 使用RecursiveCharacterTextSplitter将长文档切分成小块
    # 参数说明：
    # - chunk_size=100: 每块100字符，平衡语义完整性和检索精度
    # - chunk_overlap=20: 相邻块重叠20字符，保持上下文连贯性
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=100,               # 每块100字符
        chunk_overlap=20              # 重叠20字符
    )
    # 执行分块，返回分块后的Document列表
    texts = text_splitter.split_documents(documents)
    
    # 3. 生成Embedding并存储到FAISS
    # 创建OpenAI的Embedding模型实例
    # OpenAIEmbeddings使用text-embedding-ada-002模型（默认）
    # 模型会将文本转换为1536维的向量表示
    embeddings = OpenAIEmbeddings()
    # FAISS.from_documents()方法完成两件事：
    # 1) 将所有文本块通过Embedding模型转换为向量
    # 2) 将向量和原始文本存储到FAISS向量数据库中
    # 返回的db对象是FAISS向量数据库实例，支持相似度检索
    db = FAISS.from_documents(texts, embeddings)
    
    # 4. 创建检索器
    # as_retriever()将向量数据库转换为检索器接口
    # search_kwargs={"k": 3}表示检索最相似的3个文档
    # 这是RAG的标准配置：先召回相关文档，再送入LLM生成
    retriever = db.as_retriever(search_kwargs={"k": 3})
    
    # 5. 初始化LLM
    # 创建OpenAI LLM实例
    # temperature=0使输出更确定性、减少随机性，适合问答场景
    # 可选参数：max_tokens限制输出长度，top_p控制采样等
    llm = OpenAI(temperature=0)
    
    # 6. 创建RAG链
    # RetrievalQA是LangChain提供的检索问答链
    # 它整合了检索器和LLM，自动完成"检索->拼接Prompt->生成"的流程
    # 参数说明：
    # - llm=llm: 使用哪个LLM进行生成
    # - chain_type="stuff": 方式之一，将所有检索到的文档"塞"进Prompt
    #                       优点是简单，缺点是文档过长时可能超出上下文限制
    # - retriever=retriever: 使用哪个检索器获取上下文
    # - return_source_documents=True: 同时返回源文档，便于追溯
    qa_chain = RetrievalQA.from_chain_type(
        llm=llm,
        chain_type="stuff",
        retriever=retriever,
        return_source_documents=True    # 同时返回源文档
    )
    
    # 7. 执行问答
    # 准备用户问题
    query = "这段文本主要讲了什么？"
    # invoke()方法执行完整的RAG流程：
    # 1) 将query向量化
    # 2) 在FAISS中检索相似文档
    # 3) 拼接Prompt（包含检索到的文档和问题）
    # 4) 调用LLM生成回答
    # 返回结果是一个字典，包含：
    # - result: LLM生成的回答
    # - source_documents: 检索到的源文档列表
    result = qa_chain.invoke({"query": query})
    
    # 打印结果
    print("RAG回答:", result["result"])  # 打印LLM生成的回答
    print("源文档数量:", len(result["source_documents"]))  # 打印使用的文档数量

# 程序入口
if __name__ == "__main__":
    rag_pipeline_example()
```

## 7 常见问题

### 7.1 检索不到相关内容

**问题**：用户问题无法从知识库中检索到相关文档

**可能原因**：
- 知识库内容不完整
- Embedding模型与领域不匹配
- Chunk大小设置不合理
- 用户问题表达与知识库内容表述差异大

**解决方案**：
- 补充知识库内容
- 使用领域特定的Embedding模型
- 调整Chunk参数
- 使用查询重写、查询扩展等优化技术

### 7.2 上下文长度超限

**问题**：检索到的文档总长度超出LLM的上下文限制

**解决方案**：
- 减少检索数量（k值）
- 对检索结果进行摘要压缩
- 使用Map-Reduce等chain_type分块处理
- 对文档按重要性排序，截断不重要的部分

### 7.3 生成质量不高

**问题**：LLM生成的回答不准确或不完整

**可能原因**：
- 检索结果不相关
- Prompt设计不合理
- LLM能力不足
- 知识库质量不高

**解决方案**：
- 优化检索环节（增加重排序、使用混合检索）
- 改进Prompt设计，添加指令约束
- 使用更强大的LLM
- 提升知识库质量

### 7.4 响应延迟过高

**问题**：RAG系统响应时间过长

**解决方案**：
- 使用更快的Embedding模型
- 优化向量数据库查询性能
- 添加缓存机制
- 减少检索数量
- 使用流式输出

## 8 应用场景

### 8.1 企业知识管理

**场景**：企业部署内部知识库系统，员工查询公司文档、政策、流程等信息。

**RAG应用**：
- 加载PDF、Word等格式的内部文档
- 使用中文Embedding模型（如bge-base-zh）
- 可以结合混合检索提升召回率
- 添加权限控制，只返回有权限访问的文档

### 8.2 智能客服

**场景**：电商、金融等行业使用AI客服回答用户问题。

**RAG应用**：
- 接入产品FAQ、商品详情、售后政策等知识
- 支持多轮对话，保持上下文
- 可以针对常见问题设置缓存
- 结合意图识别，提高问答准确率

### 8.3 技术文档助手

**场景**：开发者通过问答方式检索API文档、代码示例等。

**RAG应用**：
- 支持代码块的完整检索
- 可以按编程语言、框架等维度构建知识库
- 优先返回官方文档内容
- 支持版本管理，检索对应版本文档

### 8.4 医疗健康咨询

**场景**：提供医疗健康信息咨询服务。

**RAG应用**：
- 知识库包含医学指南、药品说明、临床案例等
- 对生成内容添加免责声明
- 可以进行多文档融合，综合多篇文献回答
- 严格控制引用来源的权威性

### 8.5 法律咨询

**场景**：法律领域的文档检索和咨询。

**RAG应用**：
- 构建包含法律条文、判例、合同模板的知识库
- 支持法律专业术语的精确匹配
- 可以结合知识图谱处理法律实体关系
- 对引用内容进行溯源标注

### 8.6 学术研究辅助

**场景**：研究人员检索学术论文、专利等。

**RAG应用**：
- 接入论文数据库、专利数据库
- 支持复杂的学术检索需求
- 可以进行跨文档的综合分析
- 结合引用关系进行扩展检索
