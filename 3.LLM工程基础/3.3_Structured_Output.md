# 5.3 Structured Output

## 1 技术简介

**技术定义**：Structured Output（结构化输出）是LLM的一种能力，允许模型生成符合预定义格式（如JSON、XML）的输出。这使得LLM的输出可以被程序直接解析和处理，是构建AI应用的关键技术。

结构化输出的核心价值：
- **程序化处理**：输出可直接被代码解析，无需额外解析逻辑
- **数据一致性**：确保输出格式统一，便于后续处理
- **减少错误**：避免手动解析导致的错误
- **提升用户体验**：标准化输出便于展示和交互

**结构化输出的主要方式**：
1. **JSON Mode**：强制输出有效JSON
2. **JSON Schema**：通过Schema约束输出结构
3. **Pydantic**：使用Python数据模型定义输出格式

## 2 技术背景

**为什么需要结构化输出**：

1. **应用集成的需求**：
   - AI应用需要与其他系统对接
   - 结构化数据是系统间交互的标准
   - 非结构化输出难以进行后续处理

2. **数据质量保证**：
   - 手动解析JSON容易出错
   - 格式不一致导致代码脆弱
   - 需要在源头保证输出格式正确

3. **技术演进**：
   - 早期：依赖prompt引导，输出不可靠
   - 2023年：OpenAI引入JSON Mode
   - 2024年：引入JSON Schema和工具调用
   - 现在：主流LLM都支持结构化输出

**结构化输出vs自然语言输出**：
| 特性 | 结构化输出 | 自然语言输出 |
|------|-----------|-------------|
| 可解析性 | 高 | 低 |
| 一致性 | 高 | 低 |
| 可读性 | 一般 | 高 |
| 处理成本 | 低 | 高 |

## 3 核心概念

**JSON Mode**：
- 强制模型输出有效的JSON格式
- 不要求特定的JSON结构
- 适用于只需要JSON格式，不需要严格Schema的场景

**JSON Schema**：
- 定义输出的完整结构
- 包含字段类型、必填项、枚举值等约束
- 模型输出必须符合Schema定义

**Pydantic**：
- Python的数据验证库
- 可自动生成JSON Schema
- 支持复杂的数据验证和转换

**response_format参数**：
- 控制模型输出的格式
- `{"type": "json_object"}` - JSON模式
- `{"type": "json_schema", "json_schema": {...}}` - Schema模式

## 4 工作原理

**JSON Mode的工作原理**：
1. 设置`response_format: {"type": "json_object"}`
2. 模型在输出时会强制生成有效JSON
3. 需要在prompt中说明期望的JSON结构
4. 模型可能不严格遵守Schema约束

**JSON Schema的工作原理**：
1. 定义完整的JSON Schema
2. 通过API传递给模型
3. 模型会尽力生成符合Schema的输出
4. 在prompt中说明可以提高成功率

**Pydantic + Structured Output**：
1. 使用Pydantic定义数据模型
2. 从模型生成JSON Schema
3. 通过API传递Schema约束
4. 输出后使用Pydantic验证解析

## 5 技术实现

**使用JSON Schema的基本结构**：
```python
response_format={
    "type": "json_schema",
    "json_schema": {
        "name": "schema_name",
        "schema": {
            "type": "object",
            "properties": {...},
            "required": [...]
        }
    }
}
```

**Pydantic模型定义**：
```python
from pydantic import BaseModel, Field
from typing import List

class User(BaseModel):
    name: str = Field(description="用户名")
    age: int = Field(description="年龄")
    hobbies: List[str] = Field(description="爱好列表")
```

**结构化输出的最佳实践**：
1. 在prompt中明确描述期望的结构
2. 使用JSON Schema进行强约束
3. 输出后进行验证和错误处理
4. 提供足够的上下文帮助模型理解

class User(BaseModel):
    """用户数据模型：定义期望的输出结构"""
    name: str = Field(description="用户的姓名")           # 姓名字段
    age: int = Field(description="用户的年龄")            # 年龄字段
    hobbies: List[str] = Field(description="用户的爱好列表")  # 爱好列表

def structured_output_example():
    """结构化输出示例
    
    强制模型输出符合预定义格式的JSON数据
    """
    # 提示模型生成用户信息
    prompt = "请生成一个用户信息，包含姓名、年龄和3个爱好。"
    
    # 使用response_format约束输出格式
    response = client.chat.completions.create(
        model="gpt-3.5-turbo",
        messages=[{"role": "user", "content": prompt}],
        response_format={
            "type": "json_schema",     # 使用JSON Schema约束
            "json_schema": {
                "name": "user_schema", # Schema名称
                "schema": User.model_json_schema()  # 从Pydantic模型生成Schema
            }
        }
    )
    
    # 解析JSON响应
    user_json = json.loads(response.choices[0].message.content)
    
    # 使用Pydantic验证并解析数据
    user = User(**user_json)
    
    # 打印结构化结果
    print("结构化输出结果:")
    print(f"姓名: {user.name}")
    print(f"年龄: {user.age}")
    print(f"爱好: {', '.join(user.hobbies)}")

def pydantic_model_detail():
    """Pydantic模型详解"""
    print("""
    Pydantic是Python中用于数据验证的库，它可以根据Python类型注解自动生成JSON Schema。
    在LLM结构化输出场景中，Pydantic可以用于：
    1. 定义期望的输出结构
    2. 生成JSON Schema给LLM
    3. 验证和解析LLM输出
    
    基础模型定义：
    
    from pydantic import BaseModel, Field
    from typing import List, Optional
    
    class Person(BaseModel):
        name: str = Field(description="人名")
        age: int = Field(description="年龄", ge=0, le=150)
        email: Optional[str] = Field(description="邮箱", default=None)
        hobbies: List[str] = Field(description="爱好列表", default_factory=list)
    
    字段类型：
    
    1. 基本类型
       - str: 字符串
       - int: 整数
       - float: 浮点数
       - bool: 布尔值
    
    2. 容器类型
       - List[str]: 字符串列表
       - Dict[str, int]: 字符串到整数的映射
       - Optional[str]: 可选字符串
    
    3. 嵌套模型
       - 可以嵌套定义复杂结构
       - 支持多层嵌套
    
    Field参数：
    
    1. description: 字段描述（用于生成Schema）
    2. default: 默认值
    3. default_factory: 默认值工厂函数
    4. ge/gt/le/lt: 数值范围约束
    5. min_length/max_length: 字符串长度约束
    6. pattern: 正则表达式约束
    
    生成Schema：
    
    schema = Person.model_json_schema()
    # 返回标准的JSON Schema
    
    验证输出：
    
    # 假设llm_output是从LLM获取的JSON字符串
    data = json.loads(llm_output)
    person = Person(**data)  # 自动验证
    
    如果验证失败会抛出ValidationError异常。
    
    高级特性：
    
    1. 验证器
       @field_validator
       @classmethod
       def validate_field(cls, v):
           # 自定义验证逻辑
           return v
    
    2. 序列化器
       @model_serializer
       def to_custom_format(self):
           # 自定义序列化逻辑
           return {"custom": "format"}
    
    3. 继承
       class Employee(Person):
           employee_id: str
           department: str
    """)

def output_validation():
    """输出验证与错误处理详解"""
    print("""
    结构化输出虽然提高了可靠性，但仍可能出现各种错误。
    完善的验证和错误处理是生产环境应用的必要保障。
    
    常见错误类型：
    
    1. 解析错误
       - LLM输出不是有效JSON
       - 可能包含额外文本或markdown
    
    2. 字段错误
       - 缺少必填字段
       - 字段类型不匹配
       - 字段值超出范围
    
    3. 语义错误
       - 字段值不符合业务逻辑
       - 如：日期晚于当前日期
    
    验证策略：
    
    1. 解析验证（第一层）
       try:
           data = json.loads(output)
       except json.JSONDecodeError:
           # 处理解析失败
           output = extract_json(output)
           data = json.loads(output)
    
    2. Schema验证（第二层）
       - 使用jsonschema库验证
       - 验证字段类型和必填项
    
    3. 业务验证（第三层）
       - 使用Pydantic验证
       - 验证业务逻辑约束
    
    错误处理模式：
    
    1. 重试模式
       def get_structured_output(prompt, schema, max_retries=3):
           for attempt in range(max_retries):
               try:
                   output = call_llm(prompt, schema)
                   return validate_and_parse(output, schema)
               except ValidationError as e:
                   if attempt == max_retries - 1:
                       raise
                   # 重试，带着错误信息
                   prompt += f"\\n上次输出错误：{str(e)}"
    
    2. 回退模式
       def get_structured_output(prompt, schema):
           try:
               return strict_parse(prompt, schema)
           except Exception:
               # 回退到宽松解析
               return loose_parse(prompt)
    
    3. 修复模式
       def get_structured_output(prompt, schema):
           try:
               output = call_llm(prompt)
               data = json.loads(output)
               return fix_and_validate(data, schema)
           except Exception as e:
               return generate_default(schema)
    
    清理LLM输出的技巧：
    
    1. 去除markdown标记
       output = output.strip().strip("```json").strip("```")
    
    2. 提取JSON部分
       import re
       match = re.search(r'\\{[^{}]*\\}', output, re.DOTALL)
       if match:
           output = match.group()
    
    3. 处理转义字符
       output = output.encode().decode('unicode_escape')
    """))

def streaming_json():
    """流式JSON输出详解"""
    print("""
    在某些场景下，需要边生成边返回JSON，而不是等完整JSON生成后再返回。
    这需要特殊处理，因为JSON流式输出的中间状态可能不是有效的JSON。
    
    流式JSON的挑战：
    
    1. 中间状态无效
       - 流式输出过程中，JSON可能不完整
       - 无法直接解析
    
    2. 需要累积缓冲区
       - 需要等待完整JSON
       - 或者返回部分有效JSON
    
    3. 结构化流式困难
       - 普通流式只返回文本片段
       - 需要自己处理JSON结构
    
    解决方案：
    
    1. 等待完整JSON
       - 累积所有流式输出
       - 完整后解析
       - 简单但无实时性
    
    2. 流式解析器
       - 使用ijson库
       - 逐步解析JSON流
       - 可以处理部分JSON
    
    3. 增量流式
       - 设计支持增量的JSON结构
       - 如：JSON Lines格式
       - 每行一个完整JSON对象
    
    实现示例（累积缓冲区）：
    
    buffer = ""
    for chunk in stream:
        buffer += chunk
        # 检查是否可能构成完整JSON
        if is_valid_json_prefix(buffer):
            # 可以显示部分预览
            preview = extract_preview(buffer)
            yield {"type": "preview", "data": preview}
    
    # 最终返回完整解析结果
    result = json.loads(buffer)
    yield {"type": "complete", "data": result}
    
    实际应用场景：
    
    1. 长JSON输出
       - 大型报告生成
       - 批量数据处理
    
    2. 实时进度展示
       - 显示已生成部分
       - 用户知道处理进度
    
    3. 错误恢复
       - 流式过程中出错
       - 只需要重新生成部分
    
    注意事项：
    
    1. 缓冲区管理
       - 控制缓冲区大小
       - 防止内存溢出
    
    2. 错误检测
       - 检测JSON语法错误
       - 及时反馈
    
    3. 性能权衡
       - 流式解析有开销
       - 根据场景选择
    """)

def complex_nested_structures():
    """复杂嵌套结构详解"""

    常见复杂结构：

       {
         "user": {
           "profile": {
             "name": "张三",

           }
         }
       }

       {

           "shipping": {
             "address": "北京",
             "method": "快递"
           }
         }
       }
    
    Pydantic实现：
    
    from pydantic import BaseModel, Field

        name: str
        age: int = Field(ge=0, le=150)
        address: Address
    
    class User(BaseModel):
        user_id: str

    # 使用
    user = User.model_validate_json(json_string)

def enterprise_extraction_example():
    """企业数据提取完整示例"""
    from pydantic import BaseModel, Field, field_validator
    from typing import List, Optional
    import json
    
    # ========================================
    # 1. 定义数据模型
    # ========================================
    
    # 地址信息
    class Address(BaseModel):
        """地址信息模型"""
        province: str = Field(description="省份")
        city: str = Field(description="城市")
        district: str = Field(description="区县", default="")
        street: str = Field(description="街道详细地址")
        zip_code: Optional[str] = Field(description="邮政编码", default=None)
    
    # 联系人信息
    class Contact(BaseModel):
        """联系人信息模型"""
        name: str = Field(description="联系人姓名")
        phone: str = Field(description="联系电话")
        email: Optional[str] = Field(description="邮箱", default=None)
    
    # 商品项
    class ProductItem(BaseModel):
        """订单商品项模型"""
        product_id: str = Field(description="商品ID")
        product_name: str = Field(description="商品名称")
        quantity: int = Field(description="数量", ge=1)
        unit_price: float = Field(description="单价", ge=0)
        
        @property

## 7 常见问题

**结构化输出常见问题**：

1. **输出不是有效JSON**
   - 原因：模型输出可能包含额外文本
   - 解决：使用JSON Mode强制约束

2. **字段缺失或类型错误**
   - 原因：Schema定义不清晰或模型理解有误
   - 解决：详细描述每个字段，提供示例

3. **输出包含markdown代码块**
   - 原因：模型默认使用markdown格式
   - 解决：在prompt中说明不要使用代码块

4. **嵌套结构难以处理**
   - 原因：复杂嵌套Schema增加模型难度
   - 解决：简化结构，分步处理

5. **验证失败处理**
   - 原因：输出不符合定义的模型
   - 解决：添加fallback逻辑或重试机制

6. **流式输出无法解析**
   - 原因：流式过程中JSON不完整
   - 解决：使用累积缓冲区或ijson库

7. **特殊字符转义问题**
   - 原因：JSON中的特殊字符未正确转义
   - 解决：使用json.loads()自动处理
