# 4.2 Memory / 记忆系统

## 1 技术简介

**Agent记忆系统**是人工智能Agent架构中的核心组件，负责管理和维护Agent在运行过程中产生的各类信息。从本质上看，记忆系统模拟了人类大脑的记忆机制，使Agent能够"记住"过去的交互经历，并在后续对话中合理调用这些信息。

**主要用途**：
- 维持多轮对话的连贯性，让Agent理解对话上下文
- 存储用户的偏好设置和个人信息，提供个性化服务
- 累积业务知识，提升Agent的专业能力
- 实现跨会话的长期知识沉淀

**核心作用**：
1. **上下文维持**：使Agent能够理解和引用之前的对话内容
2. **信息复用**：避免重复询问用户已提供的信息
3. **个性化服务**：基于历史交互记忆用户偏好
4. **知识累积**：通过持续学习提升Agent能力

---

## 2 技术背景

### 2.1 对话系统的痛点

在大型语言模型（LLM）出现之前，对话系统主要依赖规则和模板，记忆能力极为有限。LLM的出现虽然带来了强大的自然语言理解能力，但其本身的记忆机制存在显著局限：

**上下文丢失问题**：
- LLM的上下文窗口虽有提升（从4K到128K tokens），但仍非无限
- 当对话长度超过窗口限制时，早期信息会被"遗忘"
- 长对话场景下，Agent无法回忆用户在会话初期提供的信息

**无状态特性**：
- 传统LLM是无状态的，每次请求都是独立处理
- 无法自动区分新会话与历史会话
- 用户每次都需要重复说明自己的背景信息

**长期记忆缺失**：
- 会话结束后，所有交互信息随之消失
- 无法实现跨会话的知识共享
- Agent无法"认识"老用户

### 2.2 记忆系统的演进

为解决上述问题，记忆系统应运而生：

| 阶段 | 特点 | 局限性 |
|------|------|--------|
| 无记忆 | 每次独立响应 | 完全无上下文 |
| 短期记忆 | 仅保留当前会话 | 会话结束即丢失 |
| 长期记忆 | 跨会话存储 | 检索效率低 |
| 混合记忆 | 分层管理 | 实现复杂 |

## 3 核心概念

### 3.1 短期记忆（Short-term Memory）

短期记忆，也称为**工作记忆**或**会话记忆**，用于存储当前会话中的交互信息。其特点包括：

- **时效性**：仅在当前会话期间有效，会话结束后清空
- **容量有限**：受限于上下文窗口大小或 token 数量
- **访问频繁**：对话过程中持续读写

**实现方式**：通常采用消息队列或列表结构，按时间顺序存储对话历史。

### 3.2 长期记忆（Long-term Memory）

长期记忆用于跨会话存储重要信息，实现真正的"记住"能力：

- **持久性**：会话结束后仍然保留，可被后续会话复用
- **容量大**：理论上可存储海量信息
- **需要检索**：需要有效的索引机制支持快速查询

**典型内容**：
- 用户个人信息（姓名、偏好、习惯）
- 业务知识（产品信息、常见问题）
- 交互历史（重要决策、关键节点）

### 3.3 上下文窗口（Context Window）

上下文窗口是指模型单次处理可接受的最大 token 数量：

- **有限性**：无论短期还是长期记忆，最终都受限于上下文窗口
- **滑动机制**：通过滑动窗口动态调整保留的内容
- **估算方法**：通常按 1 token ≈ 1.5~2 个中文字符 估算

### 3.4 记忆检索（Memory Retrieval）

当需要从长期记忆中获取信息时，需要高效的检索机制：

- **关键词检索**：基于文本匹配
- **向量检索**：基于语义相似度（最常用）
- **混合检索**：结合关键词和向量检索

### 3.5 记忆分层

现代Agent通常采用分层记忆架构：

```
┌─────────────────────────────────────────┐
│           长期记忆 (Long-term)           │
│    (向量数据库、文件存储、数据库)          │
├─────────────────────────────────────────┤
│           中期记忆 (Mid-term)            │
│     (会话摘要、关键信息提取)              │
├─────────────────────────────────────────┤
│           短期记忆 (Short-term)          │
│       (当前会话消息列表)                  │
└─────────────────────────────────────────┘
```

## 4 工作原理

### 4.1 记忆存储流程

```
用户输入 → 理解意图 → 判断存储类型 → 执行存储 → 维护窗口
```

**详细步骤**：
1. **接收输入**：获取用户的新输入或Agent的回复
2. **意图判断**：判断是否需要存储到短期记忆、长期记忆或两者
3. **执行存储**：
   - 短期记忆：追加到消息列表
   - 长期记忆：向量化后存入向量数据库
4. **窗口维护**：检查是否超出容量限制，触发遗忘机制

### 4.2 记忆检索流程

```
查询请求 → 检索策略 → 向量化 → 相似度计算 → 结果排序 → 返回结果
```

### 4.3 记忆衰减机制

为避免记忆无限增长，需要实现衰减机制：

**时间衰减**：
- 长时间未访问的记忆逐渐"淡化"
- 可通过设置TTL（Time To Live）或最后访问时间实现

**重要性衰减**：
- 低频访问的记忆被自动清理
- 定期评估记忆的重要性分数

**容量衰减**：
- 超出容量限制时，按策略删除旧记忆
- 常见策略：FIFO（先进先出）、LRU（最近最少使用）

## 5 技术实现

### 5.1 滑动窗口（Sliding Window）

滑动窗口是最基础的记忆管理策略：

**实现原理**：
- 维护一个固定大小的消息队列
- 新消息加入时，移除最旧的消息
- 可按消息数量或 token 数量控制窗口

**代码示例**：
```python
# 按消息数量滑动
while len(messages) > MAX_MESSAGES:
    messages.pop(0)  # 移除最旧的消息

# 按token数量滑动
while calculate_tokens(messages) > MAX_TOKENS:
    messages.pop(0)
```

**优点**：实现简单，易于理解
**缺点**：可能丢弃重要信息

### 5.2 摘要（Summarization）

通过 AI 生成对话摘要，压缩信息量：

**实现原理**：
1. 定期（如每N轮）或在窗口满时触发摘要
2. 将窗口内的消息发送给LLM生成摘要
3. 用摘要替换原始消息，释放空间

**优点**：保留核心信息，大幅压缩体积
**缺点**：可能丢失细节，需要额外API调用

### 5.3 向量存储（Vector Storage）

将记忆内容转为向量，存入向量数据库：

**实现步骤**：
1. **向量化**：使用embedding模型将文本转为向量
2. **存储**：存入向量数据库（Milvus、Qdrant、Chroma等）
3. **检索**：通过相似度搜索获取相关内容

**优点**：支持语义检索，扩展性强
**缺点**：需要额外的基础设施

### 5.4 混合策略

实际应用中通常组合多种策略：

```
短期记忆 → 摘要 → 长期记忆（向量存储）
    ↓
  检索时 ← 召回相关记忆 ← 注入上下文
```

## 6 Python代码实现与逐行解析

以下是一个完整的对话记忆管理系统实现：

```python
from typing import List, Dict
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class Message:
    """对话消息数据类
    
    用于存储单条对话消息的完整信息，包含角色、内容、时间戳和元数据。
    """
    role: str                         # 角色: system/user/assistant/tool
    content: str                      # 消息内容
    timestamp: datetime = field(default_factory=datetime.now)  # 创建时间，默认为当前时间
    metadata: Dict = field(default_factory=dict)  # 额外元数据，默认为空字典

class ConversationMemory:
    """对话记忆管理器
    
    管理多轮对话的历史消息，支持上下文窗口控制。
    提供消息添加、窗口维护、上下文获取等功能。
    """
    
    def __init__(self, max_messages: int = 20, max_tokens: int = 4000):
        """
        初始化对话记忆管理器
        
        参数:
            max_messages: 最大保留消息数，默认为20
            max_tokens: 最大token数估算值，默认为4000（约等于3000中文字符）
        """
        self.messages: List[Message] = []  # 存储消息的列表，初始化为空
        self.max_messages = max_messages   # 设置消息数量上限
        self.max_tokens = max_tokens       # 设置token数量上限
        self.system_prompt: str = ""       # 系统提示词，用于引导AI行为
    
    def set_system_prompt(self, prompt: str):
        """设置系统提示词
        
        系统提示词用于定义AI助手的角色和行为准则。
        
        参数:
            prompt: 系统提示词内容
        """
        self.system_prompt = prompt
    
    def add_message(self, role: str, content: str, metadata: Dict = None):
        """添加消息到历史记录
        
        创建一个新的Message对象并添加到消息列表中，
        添加后自动调用窗口维护方法确保不超出限制。
        
        参数:
            role: 消息角色 (system/user/assistant/tool)
            content: 消息内容
            metadata: 可选的元数据字典
        """
        # 创建消息对象，role指定角色，content指定内容
        # metadata默认为空字典，如果传入了则使用传入值
        message = Message(
            role=role,
            content=content,
            metadata=metadata or {}
        )
        self.messages.append(message)  # 将消息追加到列表末尾
        
        # 维护窗口大小，确保消息数量和token数不超出限制
        self._maintain_window()
    
    def _maintain_window(self):
        """维护上下文窗口
        
        内部方法，用于在添加消息后检查并调整窗口大小。
        采用两种策略：消息数量限制和token数量限制。
        """
        # 策略1：限制消息数量
        # 当消息数超过上限时，删除最早的消息
        while len(self.messages) > self.max_messages:
            # 保留最早的system消息（索引0），只删除其他角色消息
            # 这样确保系统提示始终存在
            if len(self.messages) > 1:
                self.messages.pop(0)  # 移除列表第一个元素（最早的消息）
        
        # 策略2：基于token的滑动窗口
        # 估算方式：中文字符约等于1.5个token（中文tokenizer特点）
        total_tokens = sum(len(m.content) * 1.5 for m in self.messages)
        # 当token总数超过上限且消息数足够多时，持续删除旧消息
        while total_tokens > self.max_tokens and len(self.messages) > 2:
            self.messages.pop(0)  # 删除最早的消息
            # 重新计算token总数
            total_tokens = sum(len(m.content) * 1.5 for m in self.messages)
    
    def get_context(self) -> List[Dict]:
        """获取格式化的上下文
        
        将内部的消息列表转换为API调用所需的格式。
        包含系统提示和所有历史消息。
        
        返回:
            包含role和content的字典列表，可直接用于LLM API调用
        """
        context = []  # 初始化空列表
        
        # 添加系统提示到上下文开头
        if self.system_prompt:
            context.append({
                "role": "system",
                "content": self.system_prompt
            })
        
        # 遍历所有历史消息，转换为字典格式
        for msg in self.messages:
            context.append({
                "role": msg.role,
                "content": msg.content
            })
        
        return context  # 返回构建好的上下文列表
    
    def clear(self):
        """清空记忆
        
        清除所有消息历史，用于新会话开始或用户请求重置。
        保留系统提示词设置。
        """
        self.messages = []  # 将消息列表重置为空


class ChatSession:
    """对话会话管理器
    
    整合记忆管理和LLM API调用的完整会话类。
    负责处理用户输入、调用AI模型、存储回复的完整流程。
    """
    
    def __init__(self, memory: ConversationMemory = None):
        """
        初始化对话会话
        
        参数:
            memory: 对话记忆管理器实例，如果为None则创建新实例
        """
        # 使用传入的memory或创建新的ConversationMemory实例
        self.memory = memory or ConversationMemory()
        # 注意：此处需要OpenAI API Key，实际使用需配置
        # self.client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))
    
    def chat(self, user_input: str) -> str:
        """
        进行一次对话交互
        
        流程：添加用户消息 → 获取上下文 → 调用API → 添加助手回复 → 返回结果
        
        参数:
            user_input: 用户输入的文本
            
        返回:
            AI助手的回复文本
        """
        # 步骤1：添加用户消息到记忆
        self.memory.add_message("user", user_input)
        
        # 步骤2：获取包含系统提示和历史消息的完整上下文
        messages = self.memory.get_context()
        
        # 步骤3：调用LLM API（此处为模拟调用）
        # 实际应用中需要调用 OpenAI 或其他 LLM API
        # response = self.client.chat.completions.create(
        #     model="gpt-3.5-turbo",
        #     messages=messages
        # )
        
        # 模拟AI回复
        assistant_reply = self._mock_response(user_input, messages)
        
        # 步骤4：将助手回复添加到记忆，供后续对话使用
        self.memory.add_message("assistant", assistant_reply)
        
        return assistant_reply
    
    def _mock_response(self, user_input: str, messages: List[Dict]) -> str:
        """模拟AI响应（用于演示）"""
        # 检查是否询问名字
        if "名字" in user_input:
            # 查找历史消息中用户是否提到过名字
            for msg in messages:
                if msg["role"] == "user" and "叫" in msg["content"]:
                    # 提取名字
                    content = msg["content"]
                    if "我叫" in content:
                        name = content.split("我叫")[1].strip("。")
                        return f"你叫{name}，我记得！"
            return "抱歉，我不知道你的名字。"
        
        # 简单回复
        responses = {
            "你好": "你好！有什么我可以帮助你的吗？",
            "天气": "今天天气确实不错！",
        }
        
        for key, response in responses.items():
            if key in user_input:
                return response
        
        return "我明白了"


if __name__ == "__main__":
    # 创建记忆管理器，设置最多保留10条消息
    memory = ConversationMemory(max_messages=10)
    # 设置系统提示词，定义AI角色
    memory.set_system_prompt("你是一个 helpful 的AI助手。")
    
    # 创建会话实例
    session = ChatSession(memory)
    
    # 模拟多轮对话
    print("=== 多轮对话示例 ===\n")
    
    # 定义测试输入序列
    inputs = [
        "你好，我叫张三。",
        "我叫什么名字？",
        "今天天气不错。",
        "你还记得我的名字吗？"
    ]
    
    # 遍历输入，逐个进行对话
    for user_input in inputs:
        print(f"用户: {user_input}")
        reply = session.chat(user_input)
        print(f"助手: {reply}\n")
```

**代码结构说明**：

| 类名 | 功能 | 核心职责 |
|------|------|----------|
| `Message` | 消息数据类 | 存储单条消息的完整信息 |
| `ConversationMemory` | 记忆管理器 | 管理对话历史，实现窗口控制 |
| `ChatSession` | 会话管理器 | 整合记忆与LLM API调用 |

## 7 常见问题

### 7.1 记忆容量与成本的平衡

**问题**：保留更多记忆意味着更大的上下文，进而增加API调用成本。

**解决方案**：
- 使用摘要压缩冗长对话
- 设置合理的记忆上限
- 区分需要记忆和可以遗忘的内容

### 7.2 记忆检索的准确性

**问题**：向量检索可能返回不相关内容，或遗漏重要信息。

**解决方案**：
- 优化embedding模型选择
- 实施重排序（Reranking）机制
- 结合关键词检索进行补充

### 7.3 隐私与安全

**问题**：记忆系统存储用户敏感信息，存在隐私风险。

**解决方案**：
- 对敏感信息进行脱敏处理
- 提供记忆清除功能
- 实现数据加密存储

### 7.4 记忆一致性

**问题**：同一用户多个会话间可能出现信息冲突。

**解决方案**：
- 实施记忆版本管理
- 新信息覆盖旧信息
- 保留冲突检测机制

### 7.5 性能问题

**问题**：大量记忆的存储和检索带来性能开销。

**解决方案**：
- 异步处理记忆读写
- 实施缓存策略
- 优化数据库查询

## 8 应用场景

### 8.1 智能客服

- 记住用户历史咨询记录
- 识别老客户，自动调取背景信息
- 跨会话的问题追踪

**典型场景**：
```
用户：我的订单什么时候发货？
客服Agent：让我查一下您最近的订单...（从记忆获取订单号）
用户：谢谢
客服Agent：不客气！（记忆存储本次对话）
```

### 8.2 个人AI助手

- 记住用户偏好和习惯
- 了解用户的背景信息（工作、兴趣、家庭）
- 提供个性化服务

**典型场景**：
```
用户：我明天下午3点有会议
助手：好的，我帮您记录。（存入长期记忆）
一周后
用户：提醒我下周安排
助手：您下周暂无日程安排，但一周前记录了一个明天下午3点的会议。
```

### 8.3 教育辅导Agent

- 记录学生的学习进度和薄弱环节
- 根据历史表现推荐学习内容
- 提供个性化的练习题

**典型场景**：
```
学生：我不理解二元一次方程
Agent：（从记忆调取历史）您上周也问过类似问题，让我用更基础的方式解释...
```

### 8.4 角色扮演与游戏NPC

- 记住与玩家的交互历史
- 发展故事情节
- 维持角色一致性

### 8.5 企业知识管理

- 累积组织内部的专业知识
- 跨部门知识共享
- 新员工快速上手

## 总结

Agent记忆系统是实现智能化、个性化交互的关键基础设施。通过短期记忆与长期记忆的协同工作，Agent能够：

1. **理解上下文**：在当前会话中保持连贯对话
2. **认识用户**：跨会话记住用户特征和偏好
3. **持续学习**：不断积累和复用知识

在实际开发中，需要根据具体场景选择合适的记忆策略，平衡功能与成本，并注意隐私保护等伦理问题。随着技术的发展，记忆系统将成为AI Agent的标准配置，为各种智能应用提供坚实的基础。
