# 5.5 RAG高级架构

## 1 技术简介

RAG（Retrieval-Augmented Generation，检索增强生成）高级架构是在基础RAG之上的进阶技术体系，旨在解决基础RAG在复杂场景下面临的挑战。基础RAG虽然能够通过检索外部知识来增强生成效果，但在面对用户查询表达不准确、检索结果相关性不足、需要多步推理等复杂问题时，往往表现不佳。

RAG高级架构通过引入多种技术手段来提升系统的整体性能，主要包括以下核心技术：

**Hybrid Search（混合检索）**：结合向量检索和关键词检索（如BM25），同时利用语义匹配的广泛性和精确匹配的准确性，显著提升检索召回率。

**Rerank（重排序）**：在初步检索后，使用更复杂的模型对候选文档进行精细化排序，将最相关的内容排在最前面。

**Query Rewrite（查询重写）**：在检索前对用户查询进行优化和改写，包括查询扩展、同义词替换、问句重构等，使查询更适合检索系统。

**Multi-hop RAG（多跳推理RAG）**：处理需要多步推理的复杂问题，通过迭代检索-推理循环，逐步收集信息直至回答问题。

**GraphRAG（图增强RAG）**：利用知识图谱增强检索和生成，通过将文档转换为图结构（实体-关系-实体），支持基于图的复杂推理和关联发现。

这些技术的组合使用，可以显著提升RAG系统在复杂场景下的检索质量和生成效果，使系统能够更好地理解和回答用户的问题。

## 2 技术背景

### 2.1 RAG架构的演进历程

RAG技术的发展经历了三个主要阶段，每个阶段都针对前一代的不足进行了针对性改进：

**基础RAG（Basic RAG）**

基础RAG是最简单的RAG架构，其工作流程为：用户输入查询 → 向量检索 → 获取Top-K文档 → 拼接上下文 → 调用大语言模型生成回答。这种架构的优点是实现简单、响应速度快，但存在明显的局限性：

- 检索召回率受限于向量模型的语义理解能力
- 用户查询表达不清晰时，检索结果往往不佳
- 无法处理需要多步推理的复杂问题
- 缺乏对结构化知识的利用能力

**高级RAG（Advanced RAG）**

高级RAG在基础RAG的基础上增加了多个优化环节，形成了更为复杂但效果更好的Pipeline：查询 → 查询重写 → 混合检索 → 重排序 → 多跳推理 → 生成。相比基础RAG，高级RAG通过以下方式改进：

- 查询重写改善用户查询的表达质量
- 混合检索结合多种检索方式的优势
- 重排序精化检索结果的排序
- 多跳推理支持复杂问题的分解和逐步求解

**GraphRAG（图增强RAG）**

GraphRAG代表了RAG技术的新发展方向，它将知识图谱与RAG深度融合：查询 → 知识图谱检索 → 子图生成 → 增强生成。GraphRAG的核心创新在于：

- 利用知识图谱表示实体和关系，支持基于结构的推理
- 能够发现非显式的关联信息
- 对复杂问题的理解和推理能力显著增强
- 生成结果具有更好的可解释性

### 2.2 技术演进驱动力

RAG架构演进的驱动力主要来自三个方面：

**实际应用需求**：随着RAG在企业知识管理、智能客服、辅助决策等场景的深入应用，对系统处理复杂问题的能力提出了更高要求。

**大语言模型能力提升**：更强大的LLM为复杂RAG架构的实现提供了技术基础，使得多步推理、精确重排序等计算密集型操作成为可能。

**向量数据库和工具链成熟**：向量检索、混合检索、重排序等技术的工程实现日趋成熟，为高级RAG架构的落地提供了保障。

## 3 核心概念

### 3.1 混合检索（Hybrid Search）

混合检索是指同时使用多种检索方法并将结果进行融合的检索策略。在RAG系统中，最常用的是向量检索与关键词检索的组合。

**向量检索（Vector Search）**：基于语义相似度的检索方式。将文档和查询都编码为向量，通过计算向量之间的余弦相似度或内积来评估相关性。向量检索的优势在于能够捕获语义相似性，即使查询与文档没有字面上的关键词匹配，只要语义相近就能被检索到。但其局限性在于对专有名词、精确匹配的召回效果不佳。

**关键词检索（Keyword Search / BM25）**：基于词频和逆文档频率的传统信息检索方法。BM25是其中最成功的算法，通过评估查询词在文档中的出现频率，同时考虑文档长度和语料库中的词频分布，计算文档与查询的相关性分数。关键词检索的优势在于对精确匹配、专有名词有很好的效果，但无法理解语义相近但用词不同的查询。

**结果融合（Result Fusion）**：将两种或多种检索方式的结果按照某种策略合并。常见融合方法包括倒数排序融合（RRF）、线性加权融合、概率分布融合等。

### 3.2 重排序（Rerank）

重排序是在初步检索后使用更精确的模型对候选文档进行重新排序的技术。初步检索阶段通常使用高效的向量检索或BM25，目的是从大规模文档库中快速筛选出候选集；重排序阶段则使用更复杂但计算成本更高的模型，对少量候选文档进行精细化的相关性评估。

**Cross-Encoder模型**：重排序的常用模型架构。与Bi-Encoder分别编码查询和文档不同，Cross-Encoder将查询和文档一起编码，能够捕获查询与文档之间的细粒度交互信息，因此相关性判断更加准确，但推理速度较慢，适合对少量候选文档进行排序。

**两阶段检索（Two-Stage Retrieval）**：结合初步检索和重排序的经典架构。第一阶段使用高效的向量检索或混合检索，快速从大规模语料中筛选出Top-N（如50-100个）候选文档；第二阶段使用Cross-Encoder对候选文档进行精确重排序，最终返回Top-K（如5-10个）结果。

### 3.3 查询重写（Query Rewrite）

查询重写是在检索前对用户原始查询进行优化和改写的技术，旨在解决用户查询表达不清晰、不适合检索系统理解的问题。

**查询扩展（Query Expansion）**：通过添加同义词、相关词等扩展原始查询，增加检索的召回率。例如，将"Python机器学习"扩展为"Python机器学习 Python编程 机器学习"等多个查询变体。

**查询分解（Query Decomposition）**：将复杂查询拆分为多个简单子查询，分别检索后再合并结果。例如，"Python和Java的区别"可以拆分为"Python的特点"和"Java的特点"两个子查询。

**HyDE（Hypothetical Document Embedding）**：一种创新的查询重写方法，通过让LLM生成一个"假设的正确答案文档"，然后用这个假设文档来进行检索。假设文档包含了可能的答案内容，能够更好地匹配相关的参考文档。

**回退查询（Backoff Query）**：当原始查询检索结果不佳时，自动使用更宽泛、更通用的查询进行检索，作为备选策略。

### 3.4 多跳推理（Multi-hop RAG）

多跳推理是处理需要多步推理才能回答的复杂问题的技术。这类问题的特点是无法从单一文档或单次检索中直接获得完整答案，需要结合多个来源的信息进行逐步推理。

**跳（Hop）的概念**：一次检索-推理过程称为一跳。多跳推理系统通过多次迭代，每次迭代基于上一步的结果生成新的查询或推理新的信息，逐步逼近最终答案。

**迭代检索**：每一步检索基于前一步得到的中间信息生成新的查询，不断丰富信息集。

**中间答案生成**：每一步检索后，系统会生成一个中间答案或总结，作为下一步推理的输入。

### 3.5 GraphRAG（图增强RAG）

GraphRAG将知识图谱技术与RAG深度融合，通过图结构来表示和组织知识，增强系统的推理能力和可解释性。

**知识图谱（Knowledge Graph）**：用图结构表示知识的数据结构，主要包含实体（Entity）和关系（Relation）两类节点。实体表示现实世界中的具体对象或概念，关系表示实体之间的各种关联。

**实体-关系-实体模式**：知识图谱的基本表示模式，例如"Python" - "应用于" -> "机器学习"表示Python语言应用于机器学习领域。

**子图检索**：根据查询中的实体，从知识图谱中检索相关的子图结构，获取与查询相关的实体及其邻居节点的信息。

**图增强生成**：将知识图谱检索得到的结构化信息与文本检索结果一起，作为上下文提供给生成模型，使生成结果更加准确和具有可解释性。

## 4 工作原理

### 4.1 混合检索工作原理

混合检索的核心在于同时利用向量检索和关键词检索的优势，并通过合理的融合策略将结果合并。其工作流程如下：

**第一步：并行检索**
系统同时发起向量检索和关键词检索两个请求。向量检索将查询编码为向量，在向量空间中寻找语义最相似的文档；关键词检索计算查询中每个词在文档中的BM25得分。这两种检索方式是并行的，可以同时进行。

**第二步：分数归一化**
由于向量检索的相似度分数和BM25分数处于不同的数值范围，需要对它们进行归一化处理，使其具有可比性。常见的归一化方法包括Min-Max归一化、Z-Score归一化等。

**第三步：结果融合**
将归一化后的分数按照某种策略融合成最终的排序分数。常用的融合方法有：

- **倒数排序融合（RRF）**：不考虑原始分数的绝对值，只根据文档在不同检索结果中的排名来计算融合分数。公式为：score = Σ 1/(k+rank)，其中k是平滑参数（通常取60）。RRF的优点是不需要调参，对分数尺度不敏感。

- **线性加权融合**：将不同检索方式的结果按照预设的权重进行加权求和。公式为：score = w1×vector_score + w2×keyword_score。权重通常通过实验确定。

- **概率分布融合**：将分数归一化为概率分布，然后使用乘法或加法进行融合。这种方法能够更好地处理不同检索方法结果分布的差异。

**第四步：排序输出**
根据融合后的分数对所有候选文档进行排序，返回Top-K个结果。

### 4.2 重排序工作原理

重排序的核心思想是先粗后精，先用高效但精度较低的方法筛选出候选集，再用精确但计算成本较高的方法对候选集进行精细排序。

**第一阶段：候选生成**
使用向量检索或BM25等高效方法从大规模文档库中快速筛选出Top-N（如50-100个）候选文档。这一阶段的目标是保证召回率，尽可能多地包含相关文档，同时控制计算成本。

**第二阶段：精确排序**
使用Cross-Encoder模型对候选文档进行重排序。Cross-Encoder的输入是查询-文档对，输出是相关性分数。由于需要同时编码查询和文档，Cross-Encoder能够捕获查询和文档之间的细粒度交互，因此相关性判断比单独编码的Bi-Encoder更准确。

**分数融合（可选）**
有时会将重排序分数与初始检索分数进行加权融合，得到最终的排序分数。这种做法可以结合两种分数的优势。

### 4.3 查询重写工作原理

查询重写的目标是使原始查询更适合检索系统理解和处理。主要策略包括：

**查询扩展**
构建同义词词典或使用词向量模型找到与查询词语义相关的扩展词，将原始查询与扩展词结合形成新的查询。例如，"AI"可以扩展为"AI 人工智能 Artificial Intelligence 机器学习"等。

**查询分解**
分析查询的结构，将其拆分为多个可以独立检索的子查询。拆分的依据包括：
- 并列关系："A和B"拆分为"A"和"B"
- 因果关系：拆分为"原因"和"结果"
- 比较关系：拆分为"对比项1"和"对比项2"

每个子查询独立检索，然后将结果合并。

**HyDE方法**
让LLM根据原始查询生成一个"假设的正确答案"文档，然后用这个假设文档进行检索。假设文档包含了可能出现在真实参考文档中的关键词和表述方式，能够提高检索的匹配度。

### 4.4 多跳推理工作原理

多跳推理通过迭代的方式处理复杂问题，每一步都在前一步的基础上进一步深入。

**第一步：问题分析**
分析初始查询，判断是否为需要多跳推理的复杂问题。判断依据包括：问题是否包含多个实体、是否涉及因果关系、是否需要比较多个对象等。

**第二步：迭代检索与推理**
对于每个跳次：
1. 根据当前查询进行检索，获取相关文档
2. 从文档中提取关键信息（实体、关系、事实）
3. 基于提取的信息生成中间答案或总结
4. 根据中间答案构造下一步的查询

**第三步：答案整合**
收集所有跳次获得的信息，整合成最终答案。如果需要，可以再次调用LLM进行总结和提炼。

### 4.5 GraphRAG工作原理

GraphRAG通过知识图谱增强RAG的检索和推理能力。

**知识图谱构建**
将原始文档转换为知识图谱结构：
1. 命名实体识别（NER）：从文档中提取实体
2. 关系抽取：识别实体之间的关系
3. 图谱构建：将实体和关系存储为图结构

**图增强检索**
1. 实体识别：从查询中识别关键实体
2. 子图检索：根据实体在知识图谱中检索相关子图
3. 路径扩展：沿关系边扩展，检索多跳邻居
4. 信息整合：合并图谱检索结果和文本检索结果

**增强生成**
将图谱检索得到的结构化信息（实体、关系、子图）与文本检索结果一起作为上下文，提供给生成模型，帮助模型产生更准确、更具可解释性的回答。

## 5 技术实现

### 5.1 混合检索实现要点

**向量检索实现**

向量检索的核心是将文档和查询转换为向量表示。常用方案包括：
- 使用Sentence-Transformers等预训练模型进行编码
- 使用专门的向量数据库（如Milvus、Qdrant、Chroma）存储和检索向量
- 计算余弦相似度或内积来评估相关性

**关键词检索实现**

BM25是实现关键词检索的经典算法：
- 对文档进行分词
- 计算每个词的TF-IDF相关分数
- 根据BM25公式计算文档与查询的相关性

开源实现包括rank_bm25、pyserini等。

**融合策略选择**

选择融合策略需要考虑：
- RRF：适用于结果质量差异较大的情况，无需调参
- 线性融合：适用于需要精细控制不同检索方法权重的场景
- 概率融合：适用于需要考虑分数分布特性的场景

### 5.2 重排序实现要点

**模型选择**

常用的Cross-Encoder模型包括：
- cross-encoder/ms-marco-MiniLM-L-6-v2：轻量级，效果好
- cross-encoder/ms-marco-MiniLM-L-12-v2：更大更准确
- cross-encoder/ms-marco-Multi-Lingual-MiniLM：支持多语言

**两阶段Pipeline设计**

实现时需要注意：
- 第一阶段返回足够的候选文档（通常50-100个）
- 第二阶段重排序的文档数量不宜过多（通常5-10个）
- 可以选择性地融合初始分数和重排序分数

### 5.3 查询重写实现要点

**同义词词典构建**

可以基于：
- 人工构建的领域词典
- 词向量模型自动发现相似词
- 大语言模型生成扩展词

**分解策略**

查询分解可以基于：
- 规则：检测"和"、"与"、"vs"等连接词
- LLM：让大模型分析查询结构并进行分解
- 训练专门的分解模型

### 5.4 多跳推理实现要点

**跳数控制**

需要合理设置最大跳数：
- 跳数过少：无法处理足够复杂的问题
- 跳数过多：增加计算成本，可能导致信息冗余
- 通常设置2-4跳

**查询生成**

每一步的新查询生成可以：
- 基于提取的实体构造
- 基于中间答案扩展
- 使用LLM生成

### 5.5 GraphRAG实现要点

**知识图谱构建**

可以从以下来源构建知识图谱：
- 原始文档：通过NER和关系抽取得到
- 结构化数据：数据库、API返回的数据
- 外部知识库：已有的知识图谱

**图数据库选择**

常用图数据库：
- Neo4j：功能最完善，社区活跃
- TuGraph：国产高性能图数据库
- NebulaGraph：分布式架构，支持大规模图存储

```python
Hybrid Search实现

        self.vector_weight = vector_weight  # 保存向量检索权重
        self.keyword_weight = keyword_weight  # 保存关键词检索权重
        
        # 初始化向量模型，加载预训练的Sentence-Transformers模型
        print("加载向量模型...")
        self.encoder = SentenceTransformer('all-MiniLM-L6-v2')  # 使用轻量级模型all-MiniLM-L6-v2
        self.document_embeddings = self.encoder.encode(documents)  # 将所有文档编码为向量
        
        # 初始化BM25，构建BM25索引
        print("构建BM25索引...")
        # 对每个文档进行分词（转小写后按空格分割）
        tokenized_docs = [doc.lower().split() for doc in documents]
        self.bm25 = BM25Okapi(tokenized_docs)  # 创建BM25对象
        
        print(f"索引构建完成: {len(documents)} 个文档")  # 打印完成信息
    
    def vector_search(self, query: str, top_k: int = 10) -> List[Tuple[int, float]]:
        """
        向量检索
        
        参数:
            query: 查询文本
            top_k: 返回数量
        返回:
            [(文档索引, 相似度), ...]
        """
        # 将查询编码为向量
        query_embedding = self.encoder.encode([query])[0]
        
        # 计算余弦相似度：使用点积除以两个向量的模的乘积
        # np.dot计算点积，np.linalg.norm计算向量的L2范数（模）
        similarities = np.dot(self.document_embeddings, query_embedding) / (
            np.linalg.norm(self.document_embeddings, axis=1) *  # 计算所有文档向量的模
            np.linalg.norm(query_embedding)  # 计算查询向量的模
        )
        
        # 获取相似度最高的top_k个文档索引
        # np.argsort返回升序排列的索引，[::-1]反转实现降序，[:top_k]取前k个
        top_indices = np.argsort(similarities)[::-1][:top_k]
        
        # 将结果转换为列表格式：[(索引, 相似度), ...]
        return [(int(idx), float(similarities[idx])) for idx in top_indices]
    
    def keyword_search(self, query: str, top_k: int = 10) -> List[Tuple[int, float]]:
        """
        关键词检索（BM25）
        
        参数:
            query: 查询文本
            top_k: 返回数量
        返回:
            [(文档索引, 分数), ...]
        """
        # 对查询进行分词
        tokenized_query = query.lower().split()
        
        # 使用BM25计算每个文档与查询的相关性分数
        scores = self.bm25.get_scores(tokenized_query)
        
        # 获取分数最高的top_k个文档索引
        top_indices = np.argsort(scores)[::-1][:top_k]
        
        # 归一化分数到0-1范围：除以最大分数
        max_score = max(scores) if max(scores) > 0 else 1  # 防止除零
        
        return [(int(idx), float(scores[idx] / max_score)) for idx in top_indices]
    
    def hybrid_search(
        self,
        query: str,
        top_k: int = 10,
        fusion_method: str = "rrf"  # rrf/linear/distribution
    ) -> List[Dict]:
        """
        混合检索
        
        参数:
            query: 查询文本
            top_k: 返回数量
            fusion_method: 融合方法（rrf/linear/distribution）
        返回:
            检索结果列表
        """
        # 分别进行向量检索和关键词检索，检索更多候选以保证融合后有足够结果
        vector_results = self.vector_search(query, top_k=top_k * 2)
        keyword_results = self.keyword_search(query, top_k=top_k * 2)
        
        # 根据指定的融合方法融合结果
        if fusion_method == "rrf":
            # 倒数排序融合
            fused_scores = self._reciprocal_rank_fusion(vector_results, keyword_results)
        elif fusion_method == "linear":
            # 线性加权融合
            fused_scores = self._linear_fusion(vector_results, keyword_results)
        else:
            # 概率分布融合
            fused_scores = self._distribution_fusion(vector_results, keyword_results)
        
        # 构建最终结果列表
        results = []
        # 按融合分数降序排序并取top_k个
        for doc_idx, score in sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)[:top_k]:
            results.append({
                "index": doc_idx,  # 文档索引
                "content": self.documents[doc_idx],  # 文档内容
                "score": score,  # 融合后的总分
                "vector_score": next((s for i, s in vector_results if i == doc_idx), 0),  # 向量检索分数
                "keyword_score": next((s for i, s in keyword_results if i == doc_idx), 0)  # 关键词检索分数
            })
        
        return results
    
    def _reciprocal_rank_fusion(
        self,
        vector_results: List[Tuple[int, float]],
        keyword_results: List[Tuple[int, float]],
        k: int = 60
    ) -> Dict[int, float]:
        """
        倒数排序融合（RRF）
        
        RRF公式: score = Σ 1 / (k + rank)
        k=60是经验值，用于平滑低排名项的影响
        
        参数:
            vector_results: 向量检索结果
            keyword_results: 关键词检索结果
            k: 常数参数
        返回:
            融合后的分数字典
        """
        scores = {}  # 初始化分数字典
        
        # 处理向量检索结果：根据排名计算RRF分数
        for rank, (doc_idx, _) in enumerate(vector_results):
            # 累加RRF分数：1/(k+rank)，排名越高分数越高
            scores[doc_idx] = scores.get(doc_idx, 0) + 1.0 / (k + rank)
        
        # 处理关键词检索结果：同样方式累加
        for rank, (doc_idx, _) in enumerate(keyword_results):
            scores[doc_idx] = scores.get(doc_idx, 0) + 1.0 / (k + rank)
        
        return scores
    
    def _linear_fusion(
        self,
        vector_results: List[Tuple[int, float]],
        keyword_results: List[Tuple[int, float]]
    ) -> Dict[int, float]:
        """线性加权融合"""
        scores = {}
        
        # 将向量检索分数按权重累加
        for doc_idx, score in vector_results:
            scores[doc_idx] = scores.get(doc_idx, 0) + score * self.vector_weight
        
        # 将关键词检索分数按权重累加
        for doc_idx, score in keyword_results:
            scores[doc_idx] = scores.get(doc_idx, 0) + score * self.keyword_weight
        
        return scores
    
    def _distribution_fusion(
        self,
        vector_results: List[Tuple[int, float]],
        keyword_results: List[Tuple[int, float]]
    ) -> Dict[int, float]:
        """基于分布的融合（概率乘积）"""
        scores = {}
        
        # 计算各检索结果的分数总和，用于归一化为概率
        vector_total = sum(s for _, s in vector_results)
        keyword_total = sum(s for _, s in keyword_results)
        
        # 将向量分数归一化为概率后按权重相乘
        for doc_idx, score in vector_results:
            prob = score / vector_total if vector_total > 0 else 0
            scores[doc_idx] = scores.get(doc_idx, 1.0) * (prob ** self.vector_weight)
        
        # 将关键词分数归一化为概率后按权重相乘
        for doc_idx, score in keyword_results:
            prob = score / keyword_total if keyword_total > 0 else 0
            scores[doc_idx] = scores.get(doc_idx, 1.0) * (prob ** self.keyword_weight)
        
        return scores
```

```python
"""
Rerank模型实现
使用Cross-Encoder进行精确重排序
"""

from typing import List, Dict  # 导入类型提示
import numpy as np  # 导入numpy用于数值计算

class Reranker:
    """重排序器
    
    使用Cross-Encoder模型对初步检索结果进行精确重排序
    """
    
    def __init__(self, model_name: str = "cross-encoder/ms-marco-MiniLM-L-6-v2"):
        """
        初始化重排序器
        
        参数:
            model_name: 交叉编码器模型名称
        """
        try:
            # 尝试导入Cross-Encoder模型
            from sentence_transformers import CrossEncoder
            self.model = CrossEncoder(model_name)  # 加载预训练的Cross-Encoder模型
            self.available = True  # 标记模型可用
        except ImportError:
            # 如果导入失败，打印警告并使用备用方法
            print("警告: sentence-transformers未安装，使用简单重排序")
            self.available = False
    
    def rerank(
        self,
        query: str,
        documents: List[str],
        initial_scores: List[float] = None,
        top_k: int = 5
    ) -> List[Dict]:
        """
        重排序文档
        
        参数:
            query: 查询文本
            documents: 候选文档列表
            initial_scores: 初始检索分数（可选）
            top_k: 返回数量
        返回:
            重排序后的结果
        """
        # 空文档列表直接返回空结果
        if not documents:
            return []
        
        if self.available:
            # 使用Cross-Encoder进行精确评分
            # Cross-Encoder将查询和文档一起编码，捕获更精细的交互信息
            # 创建查询-文档对列表
            pairs = [(query, doc) for doc in documents]
            # 批量预测相关性分数
            scores = self.model.predict(pairs)
        else:
            # 备用：使用简单的关键词匹配进行重排序
            scores = self._simple_rerank(query, documents)
        
        # 如果提供了初始分数，将其与重排序分数融合
        if initial_scores and len(initial_scores) == len(documents):
            # 加权融合：重排序分数权重0.7，初始分数权重0.3
            final_scores = 0.7 * scores + 0.3 * np.array(initial_scores)
        else:
            final_scores = scores
        
        # 构建结果列表
        results = []
        for i, (doc, score) in enumerate(zip(documents, final_scores)):
            results.append({
                "index": i,  # 原始索引
                "content": doc,  # 文档内容
                "rerank_score": float(score),  # 重排序分数
                "initial_score": initial_scores[i] if initial_scores else None  # 初始分数
            })
        
        # 按重排序分数降序排序
        results.sort(key=lambda x: x["rerank_score"], reverse=True)
        
        # 返回top_k个结果
        return results[:top_k]
    
    def _simple_rerank(self, query: str, documents: List[str]) -> np.ndarray:
        """简单重排序（基于关键词匹配）"""
        # 将查询分词为词集合
        query_words = set(query.lower().split())
        scores = []
        
        for doc in documents:
            # 将文档分词为词集合
            doc_words = set(doc.lower().split())
            # 计算Jaccard相似度：交集/并集
            intersection = len(query_words & doc_words)  # 共同出现的词数
            union = len(query_words | doc_words)  # 所有不同的词数
            score = intersection / union if union > 0 else 0
            
            # 额外加分：查询词在文档开头（前100字符）出现
            for word in query_words:
                if word in doc.lower()[:100]:
                    score += 0.1
            
            # 限制分数最高为1.0
            scores.append(min(score, 1.0))
        
        return np.array(scores)


class TwoStageRetriever:
    """两阶段检索器
    
    第一阶段：快速检索（向量/BM25）获取候选集
    第二阶段：精确重排序
    """
    
    def __init__(
        self,
        hybrid_engine: 'HybridSearchEngine',
        reranker: Reranker,
        stage1_top_k: int = 50,
        stage2_top_k: int = 5
    ):
        """
        初始化两阶段检索器
        
        参数:
            hybrid_engine: 混合搜索引擎
            reranker: 重排序器
            stage1_top_k: 第一阶段返回数量
            stage2_top_k: 第二阶段返回数量
        """
        self.hybrid_engine = hybrid_engine  # 存储混合搜索引擎
        self.reranker = reranker  # 存储重排序器
        self.stage1_top_k = stage1_top_k  # 存储第一阶段top_k
        self.stage2_top_k = stage2_top_k  # 存储第二阶段top_k
    
    def retrieve(self, query: str) -> List[Dict]:
        """
        执行两阶段检索
        
        参数:
            query: 查询文本
        返回:
            最终检索结果
        """
        print(f"\n查询: {query}")
        print("=" * 50)
        
        # 第一阶段：使用混合检索快速获取候选集
        print(f"[阶段1] 混合检索 Top-{self.stage1_top_k}...")
        stage1_results = self.hybrid_engine.hybrid_search(
            query, 
            top_k=self.stage1_top_k
        )
        
        # 空结果直接返回
        if not stage1_results:
            return []
        
        # 提取文档内容和初始分数
        documents = [r["content"] for r in stage1_results]
        initial_scores = [r["score"] for r in stage1_results]
        
        print(f"  检索到 {len(documents)} 个候选文档")
        
        # 第二阶段：使用重排序器进行精确排序
        print(f"[阶段2] 重排序 Top-{self.stage2_top_k}...")
        stage2_results = self.reranker.rerank(
            query,
            documents,
            initial_scores,
            top_k=self.stage2_top_k
        )
        
        print(f"  重排序完成")
        
        return stage2_results
```

```python
"""
Query Rewrite实现
展示多种查询重写策略
"""

from typing import List, Dict  # 导入类型提示
import re  # 导入正则表达式模块

class QueryRewriter:
    """查询重写器"""
    
    def __init__(self):
        # 初始化同义词词典
        self.synonym_dict = {
            "Python": ["Python编程", "Python语言"],  # Python的同义词
            "AI": ["人工智能", "Artificial Intelligence"],  # AI的同义词
            "ML": ["机器学习", "Machine Learning"],  # ML的同义词
            "深度学习": ["神经网络", "Deep Learning"],  # 深度学习的同义词
        }
    
    def expand_query(self, query: str) -> List[str]:
        """查询扩展：添加同义词"""
        expanded = [query]  # 原始查询作为第一个元素
        # 遍历同义词词典
        for key, synonyms in self.synonym_dict.items():
            # 检查查询中是否包含词典中的关键词
            if key.lower() in query.lower():
                # 用每个同义词替换关键词，生成新的查询变体
                for syn in synonyms:
                    expanded.append(query.replace(key, syn))
        # 最多返回5个查询变体
        return expanded[:5]
    
    def decompose_query(self, query: str) -> List[str]:
        """查询分解：将复杂查询拆分为子查询"""
        # 检测并列关系（和、与、vs）
        if "和" in query or "与" in query or "vs" in query.lower():
            # 使用正则表达式按连接词分割查询
            parts = re.split(r'和|与|vs', query, flags=re.IGNORECASE)
            # 返回分割后的子查询（去除空字符串）
            return [p.strip() for p in parts if p.strip()]
        # 无需分解，返回原始查询
        return [query]
    
    def hyde_rewrite(self, query: str) -> str:
        """HyDE（假设文档嵌入）：生成假设答案"""
        # 简化为添加引导词，形成假设性的查询
        return f"关于'{query}'的详细信息"
```

### 6.4 Multi-hop RAG实现

```python
"""
Multi-hop RAG实现
支持多步推理的检索增强生成
"""

from typing import List, Dict, Optional  # 导入类型提示
from dataclasses import dataclass  # 导入dataclass装饰器

@dataclass
class ReasoningStep:
    """推理步骤数据类"""
    step_number: int  # 步骤编号
    query: str  # 当前步骤的查询
    evidence: List[str]  # 收集到的证据
    intermediate_answer: str  # 中间答案

class MultiHopRAG:
    """多跳RAG系统"""
    
    def __init__(self, retriever):
        self.retriever = retriever  # 注入检索器
        self.max_hops = 3  # 最大跳数设置为3
    
    def reason(self, initial_query: str) -> Dict:
        """多步推理"""
        steps = []  # 存储所有推理步骤
        current_query = initial_query  # 当前查询初始化为原始查询
        
        # 迭代执行多跳推理
        for hop in range(self.max_hops):
            # 使用检索器获取相关文档
            docs = self.retriever.retrieve(current_query)
            # 提取前3个文档的内容作为证据
            evidence = [d["content"] for d in docs[:3]]
            
            # 生成中间答案（此处简化处理）
            intermediate = f"第{hop+1}步找到的相关信息"
            
            # 创建推理步骤对象并添加到步骤列表
            step = ReasoningStep(
                step_number=hop + 1,
                query=current_query,
                evidence=evidence,
                intermediate_answer=intermediate
            )
            steps.append(step)
            
            # 如果不是最后一步，生成下一步的查询
            if hop < self.max_hops - 1:
                # 基于中间答案构造新的查询
                current_query = f"基于{intermediate}的后续问题"
        
        # 返回包含所有步骤和最终答案的结果
        return {
            "steps": steps,
            "final_answer": f"综合{len(steps)}步推理的结果"
        }


# 使用示例
```

### 6.5 GraphRAG实现

```python
"""
GraphRAG基础实现
展示知识图谱与RAG的结合
"""

from typing import Dict, List, Tuple, Set  # 导入类型提示
from dataclasses import dataclass  # 导入dataclass装饰器

@dataclass
class Entity:
    """实体数据类"""
    id: str  # 实体ID
    name: str  # 实体名称
    entity_type: str  # 实体类型

@dataclass
class Relation:
    """关系数据类"""
    source: str  # 源实体ID
    target: str  # 目标实体ID
    relation_type: str  # 关系类型

class KnowledgeGraph:
    """知识图谱类"""
    
    def __init__(self):
        # 初始化实体字典和关系列表
        self.entities: Dict[str, Entity] = {}
        self.relations: List[Relation] = []
    
    def add_entity(self, entity_id: str, name: str, entity_type: str):
        """添加实体到图谱"""
        # 创建实体对象并存储到字典中
        self.entities[entity_id] = Entity(entity_id, name, entity_type)
    
    def add_relation(self, source: str, target: str, relation_type: str):
        """添加关系到图谱"""
        # 创建关系对象并添加到列表
        self.relations.append(Relation(source, target, relation_type))
    
    def get_neighbors(self, entity_id: str, depth: int = 1) -> Set[str]:
        """获取邻居实体
        
        参数:
            entity_id: 实体ID
            depth: 跳数深度
        返回:
            邻居实体ID集合
        """
        neighbors = set()  # 初始化邻居集合
        current = {entity_id}  # 初始为指定实体的单点集
        
        # 按指定深度进行BFS扩展
        for _ in range(depth):
            next_level = set()  # 下一层节点
            for eid in current:
                # 遍历所有关系，寻找邻居
                for rel in self.relations:
                    # 如果当前实体是关系的源头
                    if rel.source == eid:
                        next_level.add(rel.target)
                        neighbors.add(rel.target)
                    # 如果当前实体是关系的目标
                    elif rel.target == eid:
                        next_level.add(rel.source)
                        neighbors.add(rel.source)
            current = next_level  # 移动到下一层
        
        return neighbors


class GraphRAG:
    """GraphRAG系统类"""
    
    def __init__(self, kg: KnowledgeGraph, text_retriever):
        # 注入知识图谱和文本检索器
        self.kg = kg
        self.text_retriever = text_retriever
    
    def retrieve_with_graph(
        self, 
        query: str,
        use_graph_expansion: bool = True
    ) -> List[Dict]:
        """图增强检索
        
        参数:
            query: 查询文本
            use_graph_expansion: 是否使用图扩展
        返回:
            检索结果列表
        """
        # 步骤1：执行基础文本检索
        base_results = self.text_retriever.retrieve(query)
        
        # 如果不使用图扩展，直接返回基础结果
        if not use_graph_expansion:
            return base_results
        
        # 步骤2：图增强检索
        expanded_docs = []
        
        # 遍历基础检索结果的前3个
        for result in base_results[:3]:
            # 获取结果中的实体ID（简化处理，实际应使用NER提取）
            entity_id = result.get("entity_id", "")
            
            # 如果实体存在于知识图谱中
            if entity_id in self.kg.entities:
                # 扩展获取1跳邻居实体
                neighbors = self.kg.get_neighbors(entity_id, depth=1)
                
                # 遍历邻居实体
                for neighbor_id in neighbors:
                    if neighbor_id in self.kg.entities:
                        entity = self.kg.entities[neighbor_id]
                        # 添加邻居实体信息到扩展结果
                        expanded_docs.append({
                            "content": f"相关实体: {entity.name} ({entity.entity_type})",
                            "source": "graph",
                            "entity_id": neighbor_id
                        })
        
        # 合并基础结果和扩展结果，返回最多5个扩展结果
        return base_results + expanded_docs[:5]


# 使用示例
if __name__ == "__main__":
    # 构建示例知识图谱
    kg = KnowledgeGraph()
    
    # 添加实体
    kg.add_entity("e1", "Python", "编程语言")
    kg.add_entity("e2", "机器学习", "技术领域")
    kg.add_entity("e3", "TensorFlow", "框架")
    kg.add_entity("e4", "PyTorch", "框架")
    
    # 添加关系：表示实体之间的关联
    kg.add_relation("e1", "e2", "应用于")  # Python应用于机器学习
    kg.add_relation("e2", "e3", "使用")  # 机器学习使用TensorFlow
    kg.add_relation("e2", "e4", "使用")  # 机器学习使用PyTorch
    
    # 模拟文本检索器

            return [{"content": "Python机器学习相关", "entity_id": "e1"}]
    
    # 创建GraphRAG系统并测试
```

## 7 2025-2026年RAG新进展

### 7.1 混合检索 + 重排序成为标准

截至2026年，混合检索（向量+关键词）配合Cross-Encoder重排序已成为生产级RAG系统的标准配置。单一检索方式已难以满足复杂业务场景的需求。

## 8 常见问题

### 8.1 混合检索问题

**问题：不同检索方法分数尺度不一致**

向量检索的相似度分数（通常0-1之间）和BM25分数（可能很大）不在同一尺度，直接加权会导致结果偏向某一种检索方式。

**解决方案**：在使用线性融合前进行分数归一化，如Min-Max归一化；或使用RRF等与分数尺度无关的融合方法。

**问题：权重如何设置**

向量权重和关键词权重的比例设置往往需要根据具体场景调整，没有统一的最优值。

**解决方案**：通过实验确定权重配比，可以先设置常见比例（如0.7:0.3或0.6:0.4），再根据实际效果微调；也可以使用动态权重，根据查询类型自动调整。

### 8.2 重排序问题

**问题：Cross-Encoder模型推理速度慢**

Cross-Encoder需要同时编码查询和文档对，推理速度比Bi-Encoder慢很多，不适合大规模候选集的排序。

**解决方案**：采用两阶段检索，第一阶段用高效方法筛选出较小候选集（如50-100个），再对候选集进行重排序；可以使用更轻量的Cross-Encoder模型。

**问题：重排序后结果反而变差**

有时重排序模型的判断与实际需求不符，导致重排序后的结果不如初始检索结果。

**解决方案**：保留初始分数的权重，与重排序分数进行融合；或设置重排序的触发条件，只在初始结果相关性较低时触发重排序。

### 8.3 查询重写问题

**问题：查询扩展引入噪声**

添加过多同义词或扩展词可能引入与原查询不相关的结果。

**解决方案**：限制扩展词的数量；使用与原查询语义最相关的扩展词；对扩展查询的结果设置更高的阈值。

**问题：HyDE生成假设文档质量不稳定**

HyDE方法依赖LLM生成假设文档，如果生成的文档质量不高，反而会影响检索效果。

**解决方案**：使用更强大的LLM生成假设文档；对生成的假设文档进行质量过滤；结合其他查询重写策略使用。

### 8.4 多跳推理问题

**问题：跳数设置不当**

跳数设置过少无法处理复杂问题，跳数过多增加计算成本且可能引入冗余信息。

**解决方案**：根据问题复杂度动态调整跳数；设置最大跳数上限；在每跳设置停止条件，如信息已经足够回答问题。

**问题：误差累积**

每一步检索和推理都可能引入误差，多跳情况下误差可能累积放大。

**解决方案**：在每跳进行结果验证；保留之前跳次的正确信息；在最终答案生成前进行一致性检查。

### 8.5 GraphRAG问题

**问题：知识图谱构建成本高**

从原始文档构建知识图谱需要NER和关系抽取等额外处理，建设成本较高。

**解决方案**：使用已有的外部知识图谱；使用LLM进行实体和关系抽取；只对核心文档构建图谱。

**问题：图谱与文本检索结果冲突**

知识图谱检索结果与文本检索结果可能存在冲突或不一致。

**解决方案**：设置结果优先级或加权策略；对冲突结果进行标注；让LLM在生成时综合考虑。

## 8 应用场景

### 8.1 技术选型建议

**简单问答场景**

对于事实性问答、简单问题回答等场景，基础RAG通常足够。可以考虑：
- 直接使用向量检索
- 使用BM25作为补充
- 不需要复杂的多跳推理

**需要高召回率的场景**

当检索召回率至关重要时（如知识库搜索、文档检索），推荐使用：
- Hybrid Search：结合向量检索和BM25，显著提升召回率
- 设置合适的融合方法（RRF通常效果较好）

**对结果准确性要求高的场景**

当需要将最相关的结果准确排在最前面时，推荐使用：
- Two-Stage Retrieval：第一阶段快速筛选，第二阶段精确排序
- Cross-Encoder重排序

**用户查询表达不清的场景**

当用户查询可能表述不准确、不完整时，推荐使用：
- Query Rewrite：扩展、同义词替换
- HyDE：生成假设文档辅助检索

**复杂推理场景**

当问题需要多步推理、综合多方面信息时，推荐使用：
- Multi-hop RAG：迭代检索和推理
- GraphRAG：利用知识图谱的结构化信息

### 8.2 典型应用场景

**企业知识管理系统**

企业内部的文档、知识库检索，需要处理各种格式的文档和专业术语。建议使用：
- Hybrid Search保证召回
- Query Rewrite处理用户多样化的查询表达
- GraphRAG处理需要关联多个知识点的复杂问题

**智能客服系统**

自动回答用户问题，需要快速准确地检索相关答案。建议使用：
- Hybrid Search提高召回
- Rerank确保最相关的FAQ被优先返回
- 考虑使用Query Rewrite处理口语化查询

**学术文献检索**

在学术数据库中检索相关论文，需要处理专业术语和复杂的检索需求。建议使用：
- Hybrid Search结合语义和关键词检索
- Query Rewrite进行查询扩展
- Multi-hop RAG处理需要综合多篇文献的复杂问题

**法律文档分析**

法律领域文档专业性强，需要高准确性。建议使用：
- BM25权重较高的混合检索（法律术语精确匹配重要）
- Rerank确保法律条款的准确排序
- GraphRAG处理法律条文之间的关联

**医疗健康咨询**

医疗领域对准确性要求极高，需要处理复杂的医学问题。建议使用：
- 多层次的RAG架构确保准确性
- GraphRAG处理疾病、症状、药物之间的关系
- Multi-hop RAG处理需要综合诊断的复杂案例

### 8.3 技术组合建议

**组合一：基础增强组合**
- Hybrid Search + Rerank
- 适用于大多数场景，性价比高

**组合二：高级组合**
- Query Rewrite + Hybrid Search + Rerank
- 适用于查询表达多样、结果质量要求高的场景

**组合三：复杂推理组合**
- Multi-hop RAG + GraphRAG
- 适用于需要深度推理和关联分析的场景

**组合四：完整Pipeline**
- Query Rewrite + Hybrid Search + Rerank + GraphRAG
- 适用于最复杂的生产级应用，需要较大的计算资源投入
